# Migration files and workflow

## Overview
This project uses Flask-Migrate (Alembic) to keep the database schema in sync with the
SQLAlchemy ORM models in `app.models`. The `migrations/` directory contains Alembic's
configuration and all generated migration scripts (under `migrations/versions/`).

## Why `migrations/` is in source control
The `migrations/` folder is the canonical history of schema changes. Commit it so other
contributors and deployments can reproduce and apply the exact same schema evolution.

## Key files and folders
- `migrations/` — Alembic environment and generated migration scripts.
- `migrations/env.py` — Alembic environment; configures how migrations connect to the app's DB.
- `migrations/versions/` — each file here is a migration script (one script per change).
- `scripts/migrate.py` — a helper script in the repo that automates `flask db init`, `flask db migrate`,
  and `flask db upgrade` using Flask-Migrate's Python API.

## Typical workflow
1. Update or add SQLAlchemy models in `app/models.py`.
2. Run the helper to autogenerate a migration and apply it:

```bash
python scripts/migrate.py -m "describe change briefly"
```

3. Review `migrations/versions/<rev>_*.py` and, if needed, adjust the generated operations
   (for example, add `server_default` when adding non-nullable columns to existing tables).
4. Commit both the migration script(s) and your model changes to git.
5. On deployments, run the same helper (or `flask db upgrade`) to apply migrations.

## Notes about non-nullable column additions
When adding a new column with `nullable=False` to a table that already has rows, Postgres
must be given a value for existing rows. The safe pattern used in this project is:

1. In the migration, add the column with a `server_default` (e.g. `server_default=sa.false()`
   for booleans or `server_default=sa.func.now()` for timestamps) so the DB can backfill existing rows.
2. Once backfilled, alter the column to remove the `server_default` so application-layer defaults
   control future inserts.

This preserves existing data and avoids `NOT NULL` violations when applying migrations.

## About the `scripts/migrate.py` helper
- It loads `.env` (if present) so local development `DATABASE_URL` is picked up.
- If `migrations/` does not exist it runs `flask db init` for you.
- By default it runs an autogenerate (`flask db migrate`) and then `flask db upgrade`.
- Use `--upgrade-only` to skip generating a new migration and only apply pending ones.
- Use `--init-only` to only create the `migrations/` folder.

## Safety and review
- Always inspect autogenerated migrations before committing.
- For destructive operations (dropping tables/columns) prefer manual review and backups.
- Keep `migrations/` in git and avoid regenerating history (don't delete or rewrite old version files).

## If something goes wrong
- If `flask db upgrade` fails due to data constraints, open the generated migration and add
  appropriate `server_default`/backfill SQL, re-run the upgrade, then remove the defaults.
- If you accidentally commit a wrong migration, create a new migration that fixes the issue
  rather than editing history that other collaborators may rely upon.
